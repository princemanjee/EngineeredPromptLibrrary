<PROMPT_TEMPLATE_BLUEPRINT>
  [Usage: Copy this template when designing a new prompt. Replace bracketed guidance with your content; remove meta-notes when done.]

  <SYSTEM_INSTRUCTIONS>
    [Optional] You are [role/expertise]. Your overarching mandate: [mandate or philosophy].
    Follow the structure and tags in this prompt exactly.
  </SYSTEM_INSTRUCTIONS>

  <OBJECTIVE>
    [Describe in 1–3 sentences the concrete outcome this prompt should achieve for the user. Focus on what success looks like, not on how to get there.]
  </OBJECTIVE>

  <PERSONA>
    [Optional] Describe how the AI should “show up”: expertise level, domain focus, communication style, point of view, and what assumptions it may safely make about the user.
  </PERSONA>

  <CONTEXT>
    [Optional] Provide key background the AI needs: domain, audience, existing assets, prior steps, constraints, and any relevant examples or references.
  </CONTEXT>

  <CONSTRAINTS>
    <DOS>
      [List positive requirements: must‑do behaviors, safety rules, formatting/length rules, citation needs, use of tools, etc.]
    </DOS>
    <DONTS>
      [List explicit prohibitions or boundaries: off‑limits topics, disallowed tools, forbidden styles, things not to assume or not to output.]
    </DONTS>
  </CONSTRAINTS>

  <TONE>
    [Optional] Specify desired voice and style (e.g., “concise and technical”, “friendly and explanatory, no jargon”, “formal academic with citations”).]
  </TONE>

  <TOOL_INTEGRATION>
    [Optional] List available tools/APIs and how/when to use them. Include:
    - Tool names and capabilities.
    - Typical use‑cases (what to check or fetch with each).
    - Limits or costs (rate limits, latency, accuracy caveats).
    - Exact format for tool calls or tool‑use messages.
    [If no tools are available, state that explicitly.]
  </TOOL_INTEGRATION>

  <INSTRUCTIONS>
    [Describe the concrete steps the AI should perform to achieve the <OBJECTIVE>. Use an ordered list. Be specific about operations, checks, and any domain logic. Example: gather inputs → analyze → plan → execute → verify → present.]
  </INSTRUCTIONS>

  <REASONING_GUIDANCE>
    <CHAIN_OF_THOUGHT>
      [State how the AI should reason step‑by‑step. Example:
      “Decompose complex tasks into smaller subproblems and solve them sequentially.
      Perform step‑by‑step reasoning internally; reveal intermediate reasoning only
      when the user asks for it or when it materially aids understanding.”]
    </CHAIN_OF_THOUGHT>
    <TREE_OF_THOUGHT>
      [Optional] For complex/creative problems, specify if the AI should explore multiple
      candidate approaches, briefly compare pros/cons, and then choose and justify the best path.
    </TREE_OF_THOUGHT>
    <REASONING_STEPS>
      [Optional] Specify any domain‑specific frameworks or checks the AI should use
      (e.g., legal issue spotting, scientific method, design thinking, SWOT, etc.). 
    </REASONING_STEPS>
  </REASONING_GUIDANCE>

  <ITERATIVE_PROCESS>
    1. Clarify:
       - If the task, constraints, or success criteria are ambiguous or incomplete, ask up to [N] targeted clarification questions before proceeding.
    2. Plan:
       Briefly outline your approach (steps or sections) before producing the full answer.
    3. Draft:
       - Produce an initial solution that follows <OBJECTIVE>, <INSTRUCTIONS>, <CONSTRAINTS>, and <TONE>.
       - Use tools from <TOOL_INTEGRATION> where they reduce uncertainty.
    4. Self‑Review:
       - Compare the draft against <METRICS> and <CONSTRAINTS>.
       - Identify any gaps, violations, or weak points.
    5. Refine:
       - Produce a revised, improved version that addresses all issues found in Self‑Review.
    6. Handoff:
       - Present the refined output.
       - Optionally suggest how the user can request further iterations or variations.
  </ITERATIVE_PROCESS>

  <FEW_SHOT_EXAMPLES>
    [Optional] Provide 1–3 concrete input→output pairs that demonstrate the desired behavior,
    reasoning depth, and response format. Label clearly, e.g., Example 1 (Input / Output).
  </FEW_SHOT_EXAMPLES>

  <RESPONSE_FORMAT>
    [Define the exact structure of the AI’s reply: required headings, sections, lists, code blocks,
    tables, or JSON schema. Specify which sections are mandatory vs. optional and any tagging needed.]
  </RESPONSE_FORMAT>

  <POLISH_FOR_PUBLICATION>
    [Instruct the AI to ensure the final answer is clear, coherent, and ready for the intended audience:
    - Proofread for grammar, typos, and formatting.
    - Remove internal notes or scratch work.
    - Ensure terminology is consistent.
    - Apply the requested <TONE> and <RESPONSE_FORMAT>.]
  </POLISH_FOR_PUBLICATION>

  <METRICS>
    [Define 3–7 measurable quality criteria for success (e.g., factual correctness, completeness, actionable depth, brevity, structure, creativity, adherence to tools/constraints). Optionally instruct: “Before finalizing, briefly self‑score the answer 1–5 on each metric and state one improvement you targeted in the refinement step.”]
  </METRICS>

  <FLEXIBILITY>
    [Specify precedence rules among instructions (e.g., safety > laws > <CONSTRAINTS> > <METRICS> > user preferences). Indicate how to handle conflicts or missing information, e.g.: 
    - Ask for clarification when possible. 
    - If clarification isn’t available, state assumptions explicitly and proceed conservatively.]
  </FLEXIBILITY>

  <RECAP>
    [Instruct the AI to end with a brief recap summarizing: 
    - What was done and how it aligns with <OBJECTIVE>. 
    - Key decisions or assumptions. 
    - How the user can adjust the prompt or request further iterations.]
  </RECAP>
</PROMPT_TEMPLATE_BLUEPRINT>
